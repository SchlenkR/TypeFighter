TypeFighter

TypeFighter is a small, experimental language built around a modern, inference-first type system. The headline feature is structural records instead of nominal ones: records are compared by the fields they actually have, not by a declared name — so a function that needs { name: String } accepts any record with that field, no boilerplate declarations required. On top of that: set-theoretic type syntax (|, &, { … }, literal types), row polymorphism, pattern matching with narrowing, and classical polymorphic functions — figured out by the type checker with (almost) no annotations. This site is generated from the test suite: each test is a minimal example of what the type system can — and can't yet — do. For a narrative walkthrough of what's actually implemented, see Concepts.

What's in, what's not

Every item in In today is covered by at least one passing test in this suite; items under Not yet either fail, are marked Ignore, or are absent by design. A quick comparison against concepts you'll recognize from Rust, Haskell, Scala, TypeScript, Swift, or F#:

In today

  • Structural records — matched by fields, not by name
  • Heterogeneous records — named fields and positional items ({ "Tag" & radius: Number })
  • Row polymorphism — "has at least field X"
  • Literal types — 42, "red", true each have a type of their own
  • Union types A | B — booleans are the union true | false; numeric/string literal unions for refinement
  • Record-level & — merges two record operands into one structural record at the type level
  • Discriminated unions — express Option, Result, Shape via { "Tag" & fields } | { "OtherTag" & fields } without a separate type declaration
  • Pattern matching — Expr.Match with literal, variable, and wildcard patterns; literal arms narrow the scrutinee's type
  • Polymorphic functions — fresh instantiation at every use site
  • Higher-order & partial application
  • Arrays as a built-in type constructor
  • Text parser for type expressions — TypeFighter.Parser01 reads the surface syntax from text
  • Inference with (almost) no annotations

Not yet (or absent by design)

  • AST-level let-generalization — inner let bindings don't auto-generalize the way classical Hindley–Milner does (two Ignored tests track this)
  • Record / destructuring patterns in match — only literal / var / wildcard patterns today
  • Exhaustiveness checking for match — not enforced
  • Traits / type classes / protocols — no ad-hoc polymorphism (see below)
  • Recursion — no let rec or fix-point combinator
  • Implicit conversions / subtyping — Number is never silently a String (by design)
  • Nominal types — everything is structural (by design, not a gap)
  • Effect system — effects aren't tracked

A note on Traits

In languages like Rust (traits), Haskell (type classes), Swift (protocols), or Scala (given/implicits), you can write a function like show : a -> String that works on any type a which implements a Show constraint. TypeFighter has no such mechanism — there is no way to say "any type that supports toString" or "any Num". The closest thing TypeFighter offers is row polymorphism: a function fun r -> r.name works on any record that structurally has a name field. That gives you "works on anything with field X" — but not "works on anything supporting operation X".

How to read these pages

The site has two kinds of content:

  • Concepts — a narrative reference written by hand, covering only what's actually wired up and passing tests today. No roadmap speculation lives there.
  • Test categories (listed below) — one page per feature, auto-generated from the F# test files. Every test shows its doc block (Env / Source / Inferred / Error), an optional note, and its raw F# body.

Tests marked Not yet implemented document features that are deliberately not supported today.
