Concepts

This page covers only features that are actually implemented and test-covered today. Design sketches and roadmap ideas live in docs/design/ in the repo and intentionally don't appear here — everything you read on this page corresponds to running, passing code. Each section points to the test categories that exercise it.

1. The type AST at a glance

A MonoTyp is one of six shapes — that's the entire universe of types the solver reasons about:

  • Type variable (TVar) — a placeholder that unification eventually pins down.
  • Saturated type (SaturatedTyp) — a named type applied to zero or more arguments: Number, String, Array<Number>, Map<String, Number>.
  • Function type (FunTyp) — A -> B, right-associative by convention.
  • Record type (RecordTyp) — a set of named fields and a bag of positional items. Both rows live in one record; either can be empty.
  • Literal type (LiteralTyp) — a specific number, string, or boolean value treated as a type: 42, "red", true.
  • Union type (UnionTyp) — a set of alternatives: true | false, 200 | 404 | 500, Number | String.

There is no separate IntersectionTyp: & between two record operands normalises at parse time into a single merged RecordTyp.

2. Set-theoretic type syntax

Three combinators on types, all usable from the text-level parser:

SurfaceMeaningExample
A | Bdisjunction — "is an A or a B"200 | 404 | 500
A & Bconjunction at the record level — adds B's items to A{ "Circle" & radius: Number }
{ … }record-set bracket — packages named + positional items{ x: Number & y: Number }
"foo", 42, trueliteral types — TypeScript-style"red" | "yellow" | "green"
name: Anamed property inside a record{ name: String }

Precedence. & binds tighter than | (the universal "and binds tighter than or" convention). A | B & C parses as A | (B & C). Parentheses override.

Two meanings of | around records.

  • { A } | { B } — union of two record types; the value is either a record-with-A or a record-with-B.
  • { A | B } — one record whose single slot has type A | B.

Tests: Literals, and the parser's type-expression tests in TypeFighter.Parser01.Tests/TestCases/Types.fs.

3. Structural, heterogeneous records

A record is a set of items, not a list of declared fields with a name on the envelope. Two rows live on every record:

  • Named items (properties) — x: Number, name: String.
  • Positional items — anonymous types that sit in the record's bag, order-insensitive.

Records are matched structurally: a function expecting { name: String } accepts any record that has a name field of type String, regardless of what else the record carries. No type declaration is required anywhere.

// Just a record expression — no declaration, no "class", no name.
X.MkRecord [
    X.Property "age"  (X.Lit 22)
    X.Property "name" (X.Lit "John")
]
// Inferred:  { age: Number; name: String }

Tests: Records.

4. Row polymorphism

When a function reads fields from a record argument, the inferencer collects the required fields via CHasField constraints. The argument's type closes to a record containing exactly those fields — no more, no less — so the function works on any record at least that wide.

fun r -> r.name
// Inferred:  forall a. { name: a } -> a

Pass it a { name: String; age: Number } and it works; pass it a record without name and unification fails. This is "works on anything with field X" — it is not type-class polymorphism ("works on anything supporting operation X").

Tests: Rows.

5. Pattern matching with narrowing

Expr.Match supports three pattern shapes today:

  • Literal — | 0 ->, | "yes" ->, | true ->. Emits a CHasMember constraint so the scrutinee's type closes to a union containing (at least) the matched literal.
  • Var — | y ->. Catch-all that binds the scrutinee to a name in the arm body.
  • Wildcard — | _ ->. Catch-all that adds no type constraint.

Narrowing. A match that consists only of literal arms closes the scrutinee into exactly the union of those literals:

fun s -> match s with
         | 200 -> "ok"
         | 404 -> "missing"
         | 500 -> "boom"
// Inferred:  {200 | 404 | 500} -> String

Adding a wildcard arm does not widen the scrutinee — the literal arms still determine its type. Adding a var arm also catches-all at runtime but the var is free, so the scrutinee's type is driven by the literal arms only.

Not yet: record / destructuring patterns, exhaustiveness checking.

Tests: Match.

6. Discriminated unions without type

TypeFighter has no type Shape = Circle of … | Square of … syntax, and doesn't need one. A DU is just a union of tagged records:

Option<T>   = { "None" } | { "Some" & T }
Shape       = { "Circle" & radius: Number }
            | { "Square" & side:   Number }
Result<T,E> = { "Ok"  & T } | { "Err" & E }

Each arm is a heterogeneous record whose tag is a positional literal type. Build a value of one arm with X.MkRecord [ X.Positional (X.Lit "Circle"); X.Property "radius" (X.Lit 3) ].

Matching on the tag uses the same literal-pattern machinery as numeric refinement — the wrapper record keeps its payload, the tag selects the arm. Narrowing across arm bodies (so the arm sees the right payload type) is the next step and is not wired up yet.

Tests: Composites.

7. Polymorphism

Two flavors are supported:

  • Environment-supplied polytypes. An environment entry like log : forall a. a -> Unit is instantiated fresh at every use site. Two calls get independent as.
  • Top-level generalization. An expression whose inferred type has free type variables generalises to a polytype at the top level.

Not yet: generalisation at inner let bindings. The relevant test is marked [<Ignore>] on Let bindings.

Tests: Polymorphism, Generalization, Functions.

8. Arrays

Arrays are homogeneous: all elements must unify to one element type. Mixing incompatible elements is a static error — TypeFighter does not auto-widen to Array<Number | String>.

Tests: Arrays.

9. Text parser (TypeFighter.Parser01)

A standalone parser project reads programs and type expressions from text into the same AST the tests construct by hand. Type-expression grammar (low → high precedence):

typeExpr = altTyp { "|" altTyp }
altTyp   = primTyp { "&" primTyp }
primTyp  = literalTyp | identTyp | applied | "(" typeExpr ")" | "{" recordItems "}"

Top-level & between two record operands normalises to a single merged RecordTyp at parse time; conflicting fields are rejected. Literals, built-in names (Number, String, Bool), applied types, unions, parenthesised grouping, and heterogeneous record literals are all supported.

Tests: see the TypeFighter.Parser01.Tests project in the repo.

10. What's deliberately absent

  • Nominal types. There is no way to declare type Person = { name: String } and have Person be distinct from any other record with the same fields. Structural matching is the design.
  • Implicit conversions / subtyping. Number does not silently become String. If you pass the wrong type, unification fails immediately.
  • Traits / type classes. No Show-style constraints. See the "note on Traits" on the index page.
  • Recursion. No let rec, no fix-point combinator.
  • Effects. Not tracked.
